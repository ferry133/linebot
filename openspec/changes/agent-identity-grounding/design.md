## Context

`_reason_and_act`（customer_service）組系統提示順序：`SYSTEM_PROMPT` → `memory_context`（`_recall` 的【已知規律】+【過去經驗】）→ `## 此使用者的進行中專案`（若有）。最後一段雖在最後（最強位置），但措辭軟：「當使用者提到專案時，請以這些『專案名稱』辨識與回應。」

`_recall` 注入的範例多為帶名字的查詢；`_evaluate` 僅依字數評分：`error→0.1`、`escalated→0.5`、`len>100→0.8`、`len>20→0.6`。`_reflect` 以該分數存 episode；`_recall` 將 `quality>0.7` 的 episode 標為「✓ 成功」回放。知識（knowledge）只在 `quality>=0.8 且 tools_used` 才存，故知識面已過濾無工具回答；**問題出在 episode 的品質與回放**。

實測：移除污染 episode 後，真實 `_run` 對 larryoffice 正常呼叫 query_trello 並列出 3 個專案。A/B 也顯示強化版注入穩定不問身分。

## Goals / Non-Goals

**Goals:**
- 對已知身分（role/projects 已注入）的使用者，agent 不再反問身分，直接以其專案查詢作答。
- 阻斷「反問身分的非回答被評為成功並回放」的自我增強迴圈。

**Non-Goals:**
- 不改 `_recall`/檢索演算法、不改 RBAC、不改 query_trello。
- 不在本變更重做已完成的一次性資料清理。

## Decisions

**D1：強化身分注入（主修）。**
將 `## 此使用者的進行中專案` 區塊改為明確指令，例如：
```
## 此使用者身分與專案（系統已確認，請勿再詢問他是誰）
此使用者目前進行中的專案：{names}
- 當他說「我／我的案子／我有哪些工作」時，即指上述專案。
- 直接用 query_trello 查這些專案作答，切勿反問他是誰、也不要他報名字或案場。
```
維持置於系統提示最後（在 memory_context 之後）以蓋過回放範例的偏置。

**D2：品質評分納入是否實際查詢（止血增強迴圈）。**
`_evaluate` 改為：未 `error`、未 `escalated`、且**未使用任何工具**的回答，最高給未達 0.7「成功」門檻的分（如 0.6），不論字數多長；有使用工具者維持依字數 0.6/0.8。
- 效果：反問身分（無工具）不再被存成 `quality≥0.7` 的「✓ 成功」episode，`_recall` 不再回放它為好範例。
- 取捨：純諮詢/問候等合理的無工具回答也被歸為「待加強」——可接受，因其本就不需被當「成功查詢範例」回放；知識面早已要求 `tools_used` 故不受影響。

**D3：不溯及既往的記憶清理列為前置一次性動作，不寫入程式。**
本變更只改行為（避免再生污染）；既有污染由維運清理（已做）。

## Risks / Trade-offs

- [強化注入仍被極端歷史蓋過] → 注入置於最後且語氣強制；必要時可加「忽略與此衝突的過往對話」一句。
- [無工具回答一律降分，誤傷好的 FAQ 回答] → 僅影響其在 `_recall` 的「成功」標記，不影響當下回覆品質；FAQ 知識本就走知識庫，不靠 episode 回放。
- [品質門檻調整影響其他統計] → 評分僅用於記憶權重，非對外指標；影響面侷限於記憶回放。
